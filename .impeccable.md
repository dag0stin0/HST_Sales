## Design Context

### Users
Internal PubKey ops team tracking ticket sales and revenue for HOTSTYLE III (Apr 27, 2026). They check this dashboard repeatedly during the lead-up to gauge momentum, identify payment issues, and verify guest registrations across Luma (card) and Zaprite (Bitcoin/Lightning) channels. They need to glance and know — not hunt for numbers.

### Brand Personality
**Confident, clear, understated.** This is an internal tool that respects its users' time. No flash for flash's sake — every element earns its space by making a number easier to read or a trend easier to spot. The PubKey brand warmth comes through in the color palette and subtle ambient effects, not in copy or decoration.

### Emotional Goals
1. **Instant clarity** — KPIs land in under 2 seconds of looking at the screen
2. **Quiet confidence** — the design signals "this data is reliable" through polish and restraint
3. **Momentum awareness** — progress bars, timelines, and velocity insights create a sense of forward motion without anxiety

### Aesthetic Direction
- **Dark executive dashboard** — near-black canvas (`#050505`), warm accent (`#ff3d00`), semantic colors for status
- **Typography**: Instrument Sans (clean, modern UI) + JetBrains Mono (data, metrics, timestamps)
- **Glassmorphism cards** with subtle backdrop-blur and gradient borders
- **Ambient radial gradients** for depth without distraction
- **Animated counters and progress fills** with cubic-bezier easing
- **Color coding is semantic**: blue = Luma/card, orange/BTC = Zaprite/Lightning, green = confirmed/success, gold = VIP, red = pending/error

### Anti-References
- **NO busy monitoring dashboards** — this is not Grafana or Datadog; no 40-panel walls
- **NO flat spreadsheet energy** — data should have visual hierarchy, not just rows
- **NO over-animated data viz** — counters animate once on load, then stay still

### Design Principles
1. **Glanceable over explorable.** The most important numbers (guests, revenue, days left) should be readable from 4 feet away. Detail is secondary to summary.
2. **Semantic color, not decorative color.** Every color means something — payment channel, status, tier. Never use color for pure decoration.
3. **Data density with breathing room.** Show a lot of information without feeling cramped. Generous padding inside cards, tight gaps between them.
4. **Motion earns trust.** Animated counters and progress fills signal "live data" on first load. After that, the UI is calm and stable.
5. **One file, no excuses.** Single HTML with inline CSS/JS. No build step, no dependencies beyond Google Fonts. Constraints are features.

### Technical Constraints
- Single-file HTML (inline CSS + JS), deployed on Vercel with serverless API routes
- No frameworks, no chart libraries — Canvas API for the donut chart
- Google Fonts via preconnect (Instrument Sans, JetBrains Mono)
- Responsive breakpoints: 1000px, 768px, 600px
- `prefers-reduced-motion` support, `focus-visible` outlines
- Auto-refresh every 60s via `setInterval`
- APIs: Luma (event ticketing) + Zaprite (Bitcoin/Lightning payments)
